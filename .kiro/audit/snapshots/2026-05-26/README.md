# Snapshot — 2026-05-26

Archive of state files removed during the `ui-honesty-pass` spec implementation. Preserved here for retrospective forensics in case post-hackathon analysis needs to inspect the original (broken) data.

## Contents

### `performance.json` (was `src/data/performance.json`)
NAV / performance snapshots that contained a phantom jump from $3.05 → $42.84 between consecutive snapshots (likely an MNT top-up that was incorrectly accounted for as P&L). Produced bogus metrics:

- `Sharpe: 30.92`
- `totalReturn: 1154.13%`
- `maxDrawdown: 9.13%`
- `winRate: 5%` (but using "win" semantics that don't match outcomes.json)

These values were being read by `frontend/app/api/performance/route.ts` and shown on the homepage as live performance stats. They were a misrepresentation risk.

### `loop_progress.json` (was `data/loop_progress.json`)
Last recorded cycle progress for `runBatch.js`. Contains a single entry from `2026-05-20T20:21:13Z` — 6 days stale at archive time. Used as a (false) freshness indicator.

## Why archived not deleted in commit history

These files were `gitignore`'d, so deleting them in working tree alone does not affect git history (they were never committed). However, having a copy here lets us:

- Verify what numbers the dashboard was showing before the cleanup.
- Re-derive the "agent-lifetime cumulative PnL" figure from outcomes.json and confirm the discrepancy.
- Reference exact phantom values if anyone asks why the cleanup was needed.

## Regeneration

After this archive:

- `src/data/performance.json` will be regenerated on next agent cycle by `src/metrics/performanceTracker.js` (which uses `fs.mkdirSync(..., { recursive: true })`).
- `data/loop_progress.json` will be regenerated by `src/orchestrator/runBatch.js` on next batch run.

The new files will start from a clean baseline — no phantom NAV, no stale entries.
